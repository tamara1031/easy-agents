# ADR-004: ベクターDB長期記憶 (ChromaDB)

- **ステータス**: Accepted
- **決定日**: 2026-04-25

## コンテキスト

エージェントのコンテキストウィンドウは会話をまたいで失われる。ユーザーが「覚えておいて」と言った情報や、過去の設計決定を次セッションで参照できる仕組みが必要。

## 決定

**ChromaDB + Docker によるベクターDB長期記憶**を採用する。

- バックエンド: ChromaDB 0.6.3 (Docker コンテナ)
- 埋め込みモデル: all-MiniLM-L6-v2 (ONNX, クライアントサイド)
- 距離メトリクス: コサイン類似度
- 自動テキスト分解 (段落 → センテンス, 最大500文字/チャンク)
- 自動重複排除 (類似度 ≥ 0.90 でスキップ)
- タグ分類体系 (domain/task/source/project/type)

## 結果

### メリット
- セマンティック検索で「完全一致なし」でも関連記憶を取得
- コンテナで完全自己完結 (外部クラウド不要)
- Python 標準ライブラリのみでスクリプト実装 (依存最小化)
- ワークスペース横断で共有 (`~/.local/share/copilot-memory/`)

### デメリット・リスク
- Docker 必須 (CLI エージェント環境での起動オーバーヘッド)
- `memory_search.py` がリポジトリ上でコードが途中で切れている (要修正)
- ポート 18000 が固定 (競合リスク)

## 代替案

- **ファイルベース記憶** (`MEMORY.md` 方式): 軽量だが検索精度が低い
- **外部ベクターDB** (Pinecone等): 高性能だがクラウド依存・コスト発生
- **SQLite**: 全文検索可能だがセマンティック検索なし
