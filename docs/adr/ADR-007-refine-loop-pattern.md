# ADR-007: 反復改善ループ (refine-loop) パッケージの分離

- **ステータス**: Accepted
- **決定日**: 2026-04-26

## コンテキスト

ADR-006 の Phase Gate Protocol で定義された Verify フェーズでは、当初 easy-agent 自身が REVISE ループ（自己評価）を回して成果物の品質を収束させていた。しかし以下の問題が顕在化した。

1. **作成者バイアス**: 作成者は自分の成果物を客観視できない。同一エージェントが自己レビューを繰り返しても、見落としパターンも繰り返し再生産される。
2. **REVISE 収束失敗**: AmbiguityLevel: HIGH のタスクで REVISE が 2 回連続発生してもループから抜けられず、`call-advisor` を強制発動させる Phase Gate Advisory トリガー #1 に依存していた。
3. **Fix Rule の暗黙化**: 同じクラスの修正を毎回行っているのに「これは設計問題だ」と認識する仕組みが Verify フェーズ単独では存在しなかった。
4. **再利用性**: 自己評価ループのロジックが easy-agent エージェント定義に密結合しており、他のフェーズや他のオーケストレーターから再利用できなかった。

`empirical-prompt-tuning`（mizchi/skills）が「ブランクスレートのエクゼキューターに毎回シナリオを実行させ、双方向評価でイテレーションする」原理でプロンプト品質を収束させているのを参考に、同じ原理を**汎用成果物（コード・設計・計画）の品質保証**に適用することを検討した。

## 決定

**`refine-loop` を独立 APM パッケージ**として切り出し、easy-agent の Verify フェーズの REVISE ループの代替手段として標準採用する。

### 設計の核心: 4 つの不変条件

| 不変条件 | なぜ必要か |
| :--- | :--- |
| 毎イテレーションで**新規サブエージェントを dispatch** | 文脈が混入するとバイアスが蓄積する。レビュアーは常にブランクスレートでなければならない |
| **自己評価の禁止** | refine-loop 自身がレビュアーを兼任するとループが無意味化する。`agent` ツール不可な環境では即座に ABORT |
| **`[critical]` タグはループ開始後に固定** | 動かせるゴールは収束判定を破壊する |
| **1 イテレーション = 1 テーマの修正** | 無関係な複数修正を同時に行うと、何が効いたかの因果関係が失われる |

### 収束判定 (優先順位順: ABORT > ESCALATE > MAX_ITER > CONVERGED)

| ステータス | 条件 | 対応 |
| :--- | :--- | :--- |
| `ABORT` | `[critical]` タグの追加 / 削除 / `agent` ツール不可 | エラー終了 |
| `ESCALATE` | 同一 Fix Rule が **3 回以上**出現 | 設計上の問題として呼び出し元の Phase Gate へ返す |
| `MAX_ITER` | `max_iterations` (デフォルト 3) 到達 | 残存 issues を報告して終了 |
| `CONVERGED` | `[critical]` 未達 0 件 が **2 連続** | 完了レポート出力 |

「2 連続」を採用した理由は、**収束の偶然性を排除する**ため。1 回の SUCCESS では「たまたま」その回のレビュアーが見落とした可能性を排除できない。

### Fix Rule レジャー

同一の Fix Rule が繰り返し出現するパターンを記録し、3 回出現で `ESCALATE` トリガーとする。これにより「個別の修正では解消しない設計レベルの問題」を Verify フェーズ内で囲い込まず、Phase Gate へ確実に返却する。

「同一性」は表面テキストの一致ではなく**根本原因クラスの一致**で判定する（大文字小文字無視、同義表現を統一、名詞句・ケバブケース推奨）。

### easy-agent との統合

- Verify フェーズでは自己評価 REVISE ループを使用しない。常に `refine-loop` エージェントを `agent` ツールで dispatch する。
- `refine-loop` が `CONVERGED` を返したら次フェーズ (Synthesize) へ。`MAX_ITER` を返したら残存 issues をユーザーへ報告。`ESCALATE` を返した場合のみ Implement フェーズへ戻る。
- `agent` ツールが利用不可な環境では REVISE ループ（最大 2 回）にフォールバックし、ユーザーに `[refine-loop 不可: agent ツールなし。自己評価モードで継続します]` と通知する。

## 結果

### メリット

- **作成者バイアスの構造的排除**: 毎回新規エージェントが評価するため、見落としパターンが累積しない。
- **設計問題の早期検知**: Fix Rule レジャーの 3 回出現ルールにより、「個別の修正では解消しない設計レベルの問題」を Verify フェーズで識別できる。
- **再利用性**: パッケージ化により easy-agent 以外のオーケストレーターからも呼び出せる。
- **Phase Gate Advisory との分業**: REVISE が抜けられないケースを `refine-loop` が機械的に処理し、`call-advisor` は本来の戦略判断に集中できる。

### デメリット・リスク

- **API コスト**: 毎イテレーションで新規エージェント dispatch が走るため、イテレーション数 × エージェント呼び出しコストが発生。
- **コンテキスト圧迫**: refine-loop はループ全体の文脈（Fix Rule レジャー）を保持し続けるため、長時間のループでコンテキストが膨らむ。
- **`agent` ツール依存**: 親ハーネスから `agent` ツールが提供されない環境（サブエージェントとして召喚された場合など）では機能しない。
- **`[critical]` タグの設計負担**: 呼び出し元が要件を `[critical]` / 通常要件に正しく分類する必要がある。

## 関連決定

- ADR-006: Phase Gate Protocol — Verify フェーズの REVISE 動作を `refine-loop` で置き換える基盤。
- ADR-001: Advisor パターン — `refine-loop` の `ESCALATE` 時、呼び出し元 (easy-agent) は Advisory または Parliament へ委譲する。
- mizchi/skills の `empirical-prompt-tuning` — 設計原理の起源。プロンプト品質改善の双方向評価を、汎用成果物の品質改善に拡張した。
