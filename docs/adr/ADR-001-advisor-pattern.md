# ADR-001: Advisor パターン (Executor/Advisor 分離)

- **ステータス**: Accepted
- **決定日**: 2026-04-25

## コンテキスト

複雑な判断が必要な局面で、実行エージェント (Sonnet) が全ての意思決定を担うと以下の問題が生じる:
- コンテキストウィンドウの消費増大
- 判断品質のムラ
- 不可逆な操作の誤実行リスク

## 決定

**Executor (Sonnet/Haiku) / Advisor (Opus) の役割分離**を採用する。

- **Executor**: 情報収集・実装・ツール操作を担当
- **Advisor**: 戦略判断・リスク評価・エスカレーション判定のみを担当 (read/search のみ使用可)
- 相談は7種のトリガー条件に限定 (過剰相談を防ぐ)
- Advisor はタスクあたり最大5回まで

## 結果

### メリット
- Opus の高品質判断を局所的に活用しコスト最適化 (~200-500トークン/相談)
- Executor のコンテキストを汚染せずに判断を得られる
- PROCEED/CORRECT/ESCALATE/STOP の明確な返答フォーマット

### デメリット・リスク
- 相談のオーバーヘッド (往復コスト)
- Executor が相談タイミングを誤ると効果半減
- Advisor が「STOP」判断した場合の人間介入フロー設計が必要

## 代替案

- **単一エージェントで全判断**: 実装が簡単だがコスト・品質のトレードオフ大
- **常時 Opus**: コスト過大、遅延増大
