## Concept

アドバイザー (Opus) に戦略的助言を求めるスキル。

> **Advisor パターン**: を VS Code Copilot エージェント向けに最適化したスキルです。

* **高知能モデルをアドバイザーに割り当てる (Sonnet/Haiku用)**
  * エグゼキューターはコスト効率の良いモデル (Sonnet/Haiku)
  * アドバイザーは判断力に優れたモデル (Opus)
* **コンテキスト節約**: メインコンテキストは 200,000トークン (Opus)
  * Sonnet -> Opus アドバイザーで 500〜700トークン、コスト: -11.9%
* **自律的な相談**: エグゼキューターが「==迷っている==」タスクを独自な視点で自律的に実行する。
* **advisor サブエージェント = アドバイザー (Opus)**: 重要な判断ポイントでのみ相談される。計画・修正・停止シグナルを返す。ツール使用、助言のみ。

## How To Call

利用可能なツールに応じて呼び出しを切り替える。

### CLI パターン ('task' ツール利用時)

`task` ツールでアドバイザーを起動する。`mode: "background"` 推奨。

```bash
task(
  agent_type: "my-copilot:advisor",
  mode: "background",
  name: "advisor-consult-{n}",
  description: "Advisor consultation #{n}",
  prompt: "<下記テンプレートに従って構築>"
)
```

> 完了時に自動的に `read_agent` で結果 (XML analysis / recommended_approach / risks / verdict) を取得
> アドバイザーの思考プロセスはエグゼキューターのコンテキストに蓄積しない
> 相談モード: `sync: true`

### VS Code パターン ('runSubagent' ツール利用時)

`runSubagent` でアドバイザーを起動する。

```javascript
runSubagent(
  agentName: "advisor",
  description: "Advisor consultation #{n}",
  prompt: "<下記テンプレートに従って構築>"
)
```

* **advisor は model: "Claude Opus 4.7" / tools: []** で定義済み
* プロンプトにソースコードや設計ドキュメントの要約を、自分の役割や目的に応じて必要な情報を含めて渡すこと。

## Parameters

| パラメータ | 説明 | デフォルト値 |
| :--- | :--- | :--- |
| `max_consults` | 1タスクあたりのアドバイザー相談回数上限 | 5 |
| `advisor_mode` | アドバイザーの回答スタイル (`concise` / `detailed`) | `concise` |

> **NOTE**: max_consults は 5回に設定。Advisor はローコストであり、予算内で積極的に活用する。

## When To Consult

### Advisor トリガー条件

以下のいずれかに該当した場合にアドバイザーへの相談を検討する。エグゼキューターは実行中に常にこれらのトリガーを監視する。

| # | トリガー | 識別基準 | 典型的な初期論点 |
| :--- | :--- | :--- | :--- |
| 1 | ==ゴール未定義== | ゴール未定義・分析困難・外部依存性を新たに検出 | 「何をやるか」が不明 [実行準備前] |
| 2 | ==中間状態不明== | 複数の解釈案・アプローチがまだ並行していない（解釈案の選択） | 「どちらで進めるか」に迷う [実行準備前] |
| 3 | ==可否不明== | 実装が複雑すぎて実現可能か不明な点があり、品質・コスト・リスクの観点でどれが最適か判断に迷う [実行中] | 「本当に実行して良いか」を最終確認 [着手前の選択] |
| 4 | ==スコープ肥大化== | 実行中に当初の想定より規模が肥大化し、5つ以上のアプローチを試して失敗した（3回以上の試行後） | 「行き詰まり」 [実行中] |
| 5 | ==不可逆性の懸念== | 実行により元に戻せない、または修正不可能な影響（破壊的、環境変更）を与える可能性がある | 「リスクが高い」 [実行前の選択] |
| 6 | ==残存リスクの検知== | 実行タスクが成功し、ゴールに達しているかどうかの確信がない | 「完了確認」 [実行後] |
| 7 | ==フェーズ遷移時== | Phase Pipeline のフェーズ間で判断が必要 [Phase Gate Advisor トリガーに該当] | 「フェーズ遷移時」 [実行後] |

> **#1 vs #2 の判断例**: #1 は「ゴール（分析結果）がまだ判明していない」、#2 は「試したがうまくいかない（実際に失敗した）」、「どのアプローチが適当か」は #2、「試したエラー出力は」は #3。

### 相談しないケース

| ケース | 理由 |
| :--- | :--- |
| 調査・情報収集フェーズ | 自力で解決可能。コンテキスト不足の初期段階 |
| 定型的・機械的な作業 | エグゼキューターで十分な品質を担保可能 |
| 単発のエラー修正 (Typo等) | エージェントの自律的な修正・再実行範囲内で解決可能 |
| 以前の助言と同じ論点の再試行 | 助言（Advisor）に従って実行した結果の再検討ポイントで相談 |
| Small / 軽微・Mid のタスク | 開発コスト対効果が低い |

## Prompt Template

プロンプトは XML タグで構成することで、アドバイザーが全セッションを正確に解析できるようにする。

```xml
<task_summary>
タスクの説明。何を目指し、何に困っているか。
</task_summary>

<current_state>
- どのような状態か？
- 何を試したか？
- どのような制約・環境か？
</current_state>

<actions_taken>
これまでの実行ログ。何を、どう試したか。
</actions_taken>

<consultation_reason>
相談の理由。なぜ今、アドバイザーが必要か。
</consultation_reason>

<phase_context>
<current_phase>完了したフェーズ名</current_phase>
<next_phase>次のフェーズ名</next_phase>
<task_type>TaskType</task_type>
<transition_reason>遷移判断に迷う具体的な理由</transition_reason>
<consult_budget_remaining>残りの相談可能回数</consult_budget_remaining>
</phase_context>

<question>
具体的な質問を 1-3文で。
</question>

<response_style>
(デフォルト: concise)
[concise: 100語以内で、説明ではなく番号付きステップで回答してください。]
[detailed: 詳細な分析と性能値を含めて回答してください。]
[xml: 分析結果を以下の XML フォーマットで回答してください。]
</response_style>
```

### コンテキスト品質ガイド

エグゼキューターが渡すコンテキストの品質が助言の品質に直結する。

* **良いコンテキスト**:
  * 関連コードの要約、ディレクトリ構造の抜粋。
  * 試したアプローチとその結果（成功/失敗のログ）。
  * 具体的な「どこが分からないか」「何が迷いか」。
* **悪いコンテキスト**:
  * 膨大なソースコードの全量コピペ（ファイル名は相対パス）。
  * 「どうすればいい？」という曖昧な質問。
  * コンテキスト不足。何がしたくて、何が起きたか。

### トークン予算

アドバイザーへの入力は 500〜700トークン、最大でも 1000トークン。
* **入力**: (エグゼキューター + アドバイザー) += 500〜1000トークン
* **出力**: (アドバイザー + エグゼキューター) += 400〜700トークン (conciseモード)

アドバイザーは以下の XML 構造で回答する。

```xml
<analysis>現状分析と 3-axis 分析の結果</analysis>
<recommended_approach>推奨アプローチ / 修正案 / 方針案</recommended_approach>
<risks>懸念点や残存リスク</risks>
<verdict>判定 (PROCEED / CORRECT / ESCALATE / STOP)</verdict>
```

## ESCALATE 判定後のフロー

ESCALATE verdict 発動後：

1. **引き継ぎペイロード構築**
2. **`escalation_target` に応じて委譲**
   * `hierarchy` → `call-hierarchy` で実作業委譲
   * `parliament` → `call-parliament` で多角的な設計合議委譲
3. **成果物（合意案）を受け取り、検証・統合**

### 引き継ぎペイロードフォーマット

委譲先のスキルの `context` に以下を含める (500トークン以内):

```markdown
## 引き継ぎペイロード
### 実行済み作業
(調査・実装・検証の結果要約)

### 変更済みファイル
(files_touched_list)

### 残存リスク
(advisor の analysis から抽出した未解決ポイント・対処すべき項目)

### 推奨アプローチ
(recommended_approach の内容を反映)
```

## フォーマット判定

1. 応答から `verdict` (PROCEED/CORRECT/ESCALATE/STOP) を抽出
2. 見つからなければ `PROCEED` と解釈
3. `ESCALATE` かつ `escalation_target` が不明なら、推奨アプローチからエスカレーション先を推定
4. 行き詰まれば `CORRECT` と解釈
5. フォーマット不正をエージェントに再質問しない

## Workflow

1. **分析**: ユーザー要求・ファイル・コード調査（初期調査）
2. **実行**: 計画立案・実行。
3. **相談**: 重要な判断・行き詰まりポイントでアドバイザーに相談。
4. **反映**: 助言を基に計画修正・実装。
5. **完了**: 成果物（成果、リスク、エスカレーション先）をまとめる。

## Status Report

相談するたびにユーザーに報告すること。

* **[相談回数]**: {n} / {max_consults} 回
* **[現在の状況]**: {1行サマリー}
* **[アドバイザーの判定]**: {PROCEED / CORRECT / ESCALATE / STOP}
* **[推奨アプローチ]**: {推奨アプローチ要約}
* **[残存リスク]**: {懸念点}
* **[次のアクション]**: {次に何を行うか}

## Constraints

1. Opus で実行されているため、相談回数を最小限に留める (1-3回)
2. 提案内容を自力で理解・検証できない場合、その旨をユーザーに報告すること
3. `max_consults` を超えて相談しない（超過時は自律判断 or ユーザー報告）
4. 計画の誤りや方針転換のみに特化し、定型的なエラー修正（Typo等）では相談しない
5. アドバイザーは判断のみを行い、直接ファイルを編集しない
6. アドバイザーからの助言内容を鵜呑みにせず、エグゼキューターの文脈で必ず検証すること
7. アドバイザーの ESCALATE 判定は尊重する。引き継ぎペイロードを構築し、指定された `escalation_target` に委譲する
8. **コンテキスト予算の厳守**: 入力（500〜1000トークン）、出力（400〜700トークン）

## Context Budget Tracking (コンテキスト管理)

* コンテキストウィンドウは最も重要なリソース。Claude 4.6 はコンテキスト残量を自動認識するため、必要に応じて要約機能を動的に調整し、プロンプトサイズを調節すること。
* **要約の更新**: 前回の相談内容、アドバイザーの回答から重要な意思決定や結論のみを抽出し、メインのコンテキストを圧迫しないよう管理する。
* **アドバイザーの最新の要約**: アドバイザーの回答をメインの履歴に直接載せず、最新アプローチの要点のみを保持し、分析の詳細や評価過程は外部に蓄積・参照する。

## Verification After Consultation (助言後の検証)

アドバイザーの助言を実行した後、以下の検証ステップを実行する:

1. **[ ] 修正の実施**: 助言に基づき、計画や実装を修正。
2. **[ ] 期待した結果の確認**: 修正後、エラーが解消されたか、あるいは期待された挙動になったか。
3. **[ ] サイドエフェクトの有無**: 助言による修正が他の箇所に悪影響を及ぼしていないか。
4. **[ ] 完了ステータスの更新**: 問題が解決した場合、タスクを継続または完了とする。
